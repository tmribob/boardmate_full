# Лабораторная №5: тестовая модель и минимальная реализация

## 1) Критические пользовательские сценарии
- Гость открывает защищенный профиль и перенаправляется на `/login`.
- Пользователь проходит авторизацию и получает рабочую сессию.
- Администратор пытается выполнить защищенное действие без токена (доступ запрещен).
- Пользователь получает ошибки валидации при неверных параметрах API.

## 2) Ключевые бизнес-правила
- Все защищенные backend-endpoint'ы требуют валидный Bearer-токен.
- Ограничения API-валидации должны возвращать `422`.
- Ограничения загрузки файлов: тип контента и максимальный размер.
- Сессия на frontend должна очищаться при ошибке авторизации.

## 3) Области повышенного риска
- Аутентификация/авторизация (просроченные и невалидные токены).
- Ролевой доступ к CRUD-операциям.
- Валидация входных данных на endpoint'ах.
- Обработка ошибок сессии во frontend.

## 4) Реализованные уровни тестирования

### Backend (FastAPI / pytest)
- `unit`: `test_file_service_unit.py` (валидация файлов).
- `integration`: `test_main.py` (статусы, защита endpoint'ов, валидация query/form).
- `e2e`: `test_auth_service_e2e.py` (сквозной auth-flow на уровне сервиса с моками внешних границ).

### Frontend (React / RTL + Jest)
- `unit`: `Button.test.jsx`, `AuthService.test.js`.
- `integration`: `App.test.jsx` (защита маршрута, поведение header, истечение сессии).
- `e2e (минимум)`: `App.e2e.test.jsx` (гостевой сценарий открытия защищенного роута).

## 5) Инфраструктура тестирования
- Backend: `pytest.ini` с разделением `unit/integration/e2e`.
- Frontend: `setupTests.js` для общих матчеров `jest-dom`.
- Внешние зависимости мокируются в тестах (axios/репозитории).
- Тесты изолированы и очищают локальное состояние (`localStorage.clear()`).

## 6) Метрики и запуск
- Базовое контролируемое покрытие обеспечивается по ключевым слоям (service, route guard, API-валидация).
- Разделение по уровням:
  - Backend: `pytest -m unit`, `pytest -m integration`, `pytest -m e2e`
  - Frontend: `npm test`
